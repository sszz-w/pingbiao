# Pingbiao-Power 前后端对接检查报告

基于 `pingbiao.md`（产品 spec）、`front.md`（前端开发步骤）、`back.md`（后端开发步骤）的交叉比对，发现以下**冲突与不一致**，需在开发前统一。

---

## 一、冲突清单（需修复）

### 1. `clause_start` 事件结构不一致

| 来源 | 字段 | 说明 |
|------|------|------|
| **Spec (pingbiao.md)** | `clause: {...}`, `bid_name: string` | 完整 Clause 对象 + 投标文件名 |
| **前端 types.ts** | `clause: Clause`, `bid_name: string` | 与 spec 一致 |
| **后端 back.md** | `clause_no: "1"`, `bid_id: "投标A"` | 仅条款编号 + bid_id（文件名去扩展名） |

**问题**：
- 前端 DebatePanel 需要完整 `clause`（含 `desc`、`score` 等）展示「当前条款信息」
- 前端用 `bid_name` 展示「当前投标文件名」，后端用 `bid_id`（可能为去扩展名后的文件名）
- 后端未在 `clause_start` 中下发完整 `clause` 对象

**建议**：后端在 `clause_start` 中补充：
- `clause`: 完整 Clause 对象（含 `id`, `no`, `desc`, `score`, `weight`, `order`）
- `bid_name`: 投标文件原始文件名（便于用户识别），或前后端约定 `bid_id` 即显示名

---

### 2. `debate` 事件 `content` 类型不一致

| 来源 | content 类型 | 说明 |
|------|--------------|------|
| **Spec (pingbiao.md)** | `"content": "..."` | 字符串 |
| **前端 types.ts** | `content: string` | 字符串 |
| **后端 schemas.py** | `content: dict \| None` | 对象 |

**问题**：前端 DebatePanel 需将 `content` 作为文本流式追加展示，若为 `dict` 需约定解析方式（如 `content.text` 或 `content.reason`）。

**建议**：统一为 `content: string`，后端在 yield 前将 LLM 返回的 JSON 转为可读文本（如 `reason` 字段）再下发。

---

### 3. Clause 数据模型字段不一致

| 字段 | Spec | 前端 types.ts | 后端 schemas.py |
|------|------|---------------|-----------------|
| id | ✅ | ✅ | ❌ 缺失 |
| no | ✅ | ✅ | ✅ |
| desc | ✅ | ✅ | ✅ |
| score | ✅ | ✅ | ✅ |
| weight | ✅ | ✅ | ✅ |
| order | ✅ | ✅ | ❌ 缺失 |

**问题**：后端 `Clause` 缺少 `id`、`order`，前端和 spec 均依赖；`order` 影响条款展示顺序。

**建议**：后端 `models/schemas.py` 中 Clause 补充 `id: str`、`order: int`，招标解析时生成（如 `id=f"c{i}"`，`order=i`）。

---

### 4. 前端未定义后端额外事件类型

后端会发送以下事件，前端 `StreamEvent` 类型未定义：

| 事件 | 用途 | 前端处理建议 |
|------|------|--------------|
| `parse_tender_done` | 招标解析完成，含 `clauses[]` | 可用于进度条「解析招标完成」、预加载条款列表 |
| `parse_bid_done` | 单个投标解析完成 | 可用于进度条「已解析 N 个投标」 |

**建议**：在 `front.md` 的 `types.ts` 中补充：

```ts
| { type: 'parse_tender_done'; clauses: Clause[] }
| { type: 'parse_bid_done'; bid_id: string; file_name: string }
```

并在 App/ProgressBar 中处理，用于早期进度反馈（可选，但类型需定义以免解析报错）。

---

### 5. `score` 事件字段命名

| 来源 | 字段 | 说明 |
|------|------|------|
| **Spec** | `score`, `reason` | 无 clause_no、bid_id |
| **前端** | `score`, `reason` | 与 spec 一致 |
| **后端** | `clause_no`, `bid_id`, `score`, `reason` | 多了 clause_no、bid_id |

**说明**：后端多出的 `clause_no`、`bid_id` 有助于前端归档到 `ClauseResult[]`，不冲突，可保留。前端 ScoreCard 需能关联到具体条款×投标，建议前端也消费 `clause_no`、`bid_id`（或 `bid_name`）用于分组展示。

**建议**：前后端约定 `score` 事件统一包含 `clause_no`、`bid_id`/`bid_name`、`score`、`reason`，前端 types 补充可选字段。

---

### 6. `error` 事件结构

| 来源 | 结构 |
|------|------|
| **前端** | `{ type: 'error'; message: string }` |
| **后端** | `{ type: 'error'; error: str; message: str }` |

**建议**：后端保持 `error`（错误码）、`message`（用户可读信息），前端 ErrorToast 优先展示 `message`，有 `error` 时可做分类处理。

---

## 二、一致部分（无需修改）

- **API 路径**：`POST /api/run` ✅
- **请求体**：`tender_file`, `bid_files[]`, `api_base`, `api_key`, `model` ✅
- **响应格式**：NDJSON 流 ✅
- **report 事件**：`{ type: 'report', html: string }` ✅
- **CORS / 代理**：前端 proxy `/api` → 8000，后端 CORS 允许 5173 ✅
- **文件校验**：仅 PDF，非 PDF 返回 400 + `invalid_format` ✅

---

## 三、修复建议汇总

| 优先级 | 修改方 | 修改内容 |
|--------|--------|----------|
| P0 | 后端 | `clause_start` 增加 `clause` 完整对象、`bid_name`（或约定 bid_id 即显示名） |
| P0 | 后端 | `debate` 的 `content` 改为 `string` |
| P0 | 后端 | Clause 模型补充 `id`、`order` |
| P1 | 前端 | StreamEvent 补充 `parse_tender_done`、`parse_bid_done` |
| P1 | 前端 | `score` 事件类型补充 `clause_no`、`bid_id`/`bid_name`（可选） |
| P2 | 前端 | `error` 事件支持 `error` 字段（可选） |

---

## 四、建议的统一契约（供 Claude Code 开发参考）

将以下内容同步到 `pingbiao.md` 的 12.3 节或单独「前后端契约」小节，作为开发依据：

```json

{ "type": "clause_start", "clause": { "id": "c1", "no": "1", "desc": "...", "score": 10, "weight": 1.0, "order": 1 }, "bid_name": "投标A.pdf" }

// debate
{ "type": "debate", "role": "support" | "challenge", "content": "文本内容..." }

// score
{ "type": "score", "clause_no": "1", "bid_name": "投标A.pdf", "score": 8, "reason": "..." }

// 可选：解析进度
{ "type": "parse_tender_done", "clauses": [...] }
{ "type": "parse_bid_done", "bid_id": "投标A", "file_name": "投标A.pdf" }

// error
{ "type": "error", "error": "llm_error", "message": "用户可读信息" }
```
